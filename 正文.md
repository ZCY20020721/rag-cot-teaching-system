# 基于 RAG 与 CoT 的智能教学评估系统 — 正文（中俄互译）

---

## 1.2 项目目标 / Цели проекта

### 中文正文

本毕业设计项目旨在构建一套基于检索增强生成（Retrieval-Augmented Generation, RAG）与思维链（Chain of Thought, CoT）技术的智能教学评估系统。该系统面向高等教育场景中的编程类与理论类课程，为教师和学生提供完整的教学闭环支持。具体而言，本项目设定以下四个核心目标：

**第一，实现基于教材知识的智能出题与防幻觉机制。** 传统大语言模型（Large Language Model, LLM）在直接用于教学场景时，存在严重的"幻觉"问题——即模型可能生成与教材内容无关、甚至事实性错误的题目和反馈。本系统通过 RAG 技术将 LLM 的输出锚定在教师上传的 PDF 教材之上：系统首先对教材进行文本切分与向量化存储，在出题或批改时实时检索与当前任务最相关的教材片段，将其作为上下文注入 Prompt 中，从而确保 AI 生成的题目和批改意见严格基于教材内容，从工程层面有效抑制幻觉现象。

**第二，构建具备可解释性的 CoT 分步批改引擎。** 在自动化教学评估中，单纯给出一个总分而不提供评分依据是不可接受的——学生和教师都需要理解"为什么得到这个分数"。本系统采用思维链提示工程（Chain of Thought Prompting），强制 LLM 按照"要点逐一匹配→整体逻辑评估→综合反馈"的三步推理流程进行批改。每一步的评分和评语均以结构化 JSON 格式输出，前端逐条展示，使批改过程透明、可追溯、可复核。同时，反馈采用苏格拉底式引导语，不直接透露标准答案，而是通过提问引导学生自主思考和纠正错误。

**第三，构建教师-学生双角色的完整教学业务闭环。** 系统支持教师和学生两种角色，各自拥有独立的操作权限和功能页面。教师端可上传教材构建知识库、基于 RAG 生成考题并发布、查看全体学生的答题成绩与薄弱知识点分布；学生端可浏览教师发布的习题、提交答案并获取 CoT 分步批改反馈、查看个人成绩趋势、通过错题集页面定位自身知识薄弱环节。此外，系统还提供师生实时聊天功能，支持文字、表情、图片、视频和各类文档的传输，实现课后的即时答疑与沟通。

**第四，以敏捷工程方法快速交付可用的原型系统。** 本项目采用"做减法"的工程哲学：使用 Streamlit 纯 Python 框架实现零前端开发成本、选择 SQLite 与 ChromaDB 等免安装本地数据库降低运维复杂度、调用云端 DeepSeek API 替代昂贵的本地 GPU 部署、通过 Prompt Engineering 而非模型微调来优化输出质量。整个系统从零开始至原型交付仅历时 3 天，总代码量约 1400 行 Python，充分验证了"初学者友好型"技术选型路线在 AI 教育应用开发中的可行性与高效率。

### Русский текст

Данный дипломный проект направлен на создание интеллектуальной системы оценки обучения на основе технологий поисково-дополненной генерации (Retrieval-Augmented Generation, RAG) и цепочки рассуждений (Chain of Thought, CoT). Система ориентирована на курсы программирования и теоретические дисциплины в сфере высшего образования, обеспечивая полный замкнутый цикл обучения для преподавателей и студентов. В частности, проект ставит перед собой четыре ключевые цели:

**Во-первых, реализация интеллектуальной генерации заданий на основе учебных материалов с механизмом предотвращения галлюцинаций.** Традиционные большие языковые модели (Large Language Model, LLM) при непосредственном использовании в образовательных сценариях страдают от серьёзной проблемы «галлюцинаций» — модель может генерировать задания и обратную связь, не соответствующие содержанию учебных материалов или содержащие фактические ошибки. Данная система с помощью технологии RAG привязывает вывод LLM к PDF-материалам, загруженным преподавателем: система сначала выполняет сегментацию текста и векторизацию учебных материалов, а при генерации заданий или оценивании в реальном времени извлекает наиболее релевантные фрагменты материалов и вставляет их в промпт в качестве контекста. Таким образом, генерируемые AI задания и комментарии строго основываются на содержании учебных материалов, эффективно подавляя галлюцинации на инженерном уровне.

**Во-вторых, построение объяснимого движка оценивания на основе CoT с пошаговым анализом.** В автоматизированной оценке обучения выдача лишь итогового балла без обоснования неприемлема — как студенты, так и преподаватели должны понимать, «почему получен именно такой балл». Данная система использует технику промпт-инжиниринга Chain of Thought, принуждая LLM следовать трёхшаговому процессу рассуждения: «последовательное сопоставление по критериям → общая оценка логики → комплексная обратная связь». Оценки и комментарии каждого шага выводятся в структурированном формате JSON и поэтапно отображаются на фронтенде, делая процесс оценивания прозрачным, прослеживаемым и проверяемым. При этом обратная связь использует сократический подход — вместо прямого раскрытия правильного ответа студенту задаются наводящие вопросы, побуждающие к самостоятельному размышлению и исправлению ошибок.

**В-третьих, построение полного замкнутого бизнес-цикла обучения с ролями преподавателя и студента.** Система поддерживает две роли — преподаватель и студент — каждая с собственными правами доступа и функциональными страницами. Преподаватель может загружать учебные материалы для построения базы знаний, генерировать задания на основе RAG и публиковать их, просматривать результаты ответов всех студентов и распределение слабых мест по темам. Студент может просматривать опубликованные преподавателем задания, отправлять ответы и получать пошаговую обратную связь CoT, отслеживать динамику своих оценок, выявлять слабые места через страницу «Сборник ошибок». Кроме того, система предоставляет функцию чата в реальном времени между преподавателем и студентом с поддержкой текста, эмодзи, изображений, видео и различных документов, обеспечивая оперативные консультации и коммуникацию после занятий.

**В-четвёртых, быстрая поставка работоспособного прототипа с использованием методологии гибкой разработки.** Проект следует инженерной философии «минимализма»: использование чисто Python-фреймворка Streamlit для устранения затрат на фронтенд-разработку, выбор SQLite и ChromaDB как локальных баз данных без необходимости установки и настройки, вызов облачного API DeepSeek вместо дорогостоящего локального развёртывания GPU, оптимизация качества вывода через Prompt Engineering вместо дообучения модели. Вся система была создана с нуля до прототипа всего за 3 дня, общий объём кода составляет около 1400 строк Python, что убедительно подтверждает осуществимость и высокую эффективность подхода «дружественного к начинающим» при разработке AI-приложений в образовательной сфере.

## 1.3 技术栈总览 / Обзор технологического стека

### 中文正文

本系统在技术选型上严格遵循"初学者友好、低成本、快速交付"的原则，避免引入需要复杂运维或高硬件成本的技术组件。整个系统基于 Python 生态构建，从前端到后端、从数据持久化到 AI 推理，均采用社区活跃、文档完善的成熟开源方案。以下按系统分层逐一阐述各项技术选型及其理由。

**前端层：Streamlit 1.57。** Streamlit 是一个专为数据科学和机器学习应用设计的纯 Python Web 框架。开发者无需编写任何 HTML、CSS 或 JavaScript 代码，仅通过 Python 函数调用即可构建具备表单、图表、文件上传、聊天界面等丰富交互功能的 Web 应用。对于本项目的单人开发场景，Streamlit 将前端开发成本降至几乎为零，使开发者能够将全部精力集中于核心 AI 逻辑的实现。此外，Streamlit 原生支持 `st.cache_resource` 资源缓存装饰器、`st.session_state` 会话状态管理、`st.fragment` 局部页面刷新以及 `@st.fragment(run_every=N)` 定时自动刷新机制，这些特性为聊天系统的"准实时"消息更新和多页面状态保持提供了简洁高效的实现路径。

**大模型 API 层：DeepSeek-V3。** DeepSeek-V3 是深度求索公司发布的最新⼀代大语言模型，在推理能力、数学能力和代码能力等关键维度上达到了国际一线水平。本系统选择 DeepSeek 作为模型基座主要基于三点考虑：其一，DeepSeek 的 API 定价极具竞争力（约 1 元人民币/百万 tokens），对于毕设阶段的频繁调试和大规模测试极为友好；其二，DeepSeek API 完全兼容 OpenAI SDK 接口规范，可直接使用 `langchain-openai` 包进行集成，无需额外的适配工作；其三，DeepSeek-V3 在中文和英文任务上均表现优异，能够很好地支持本系统的双语教学场景。系统调用方式为通过 `ChatOpenAI` 类（兼容接口）以 temperature=0.1 的低温度参数请求 Chat Completion，确保批改评分的一致性和可复现性。

**编排框架层：LangChain + LCEL。** LangChain 是当前 LLM 应用开发领域最主流的标准框架，提供了文档加载器（Document Loader）、文本分割器（Text Splitter）、向量存储（Vector Store）、输出解析器（Output Parser）等一系列"开箱即用"的组件。本系统主要使用 LangChain 的以下能力：`PyPDFLoader` 加载 PDF 教材、`RecursiveCharacterTextSplitter` 按语义边界切分文本、`Chroma` 封装 ChromaDB 的向量存储与检索接口、`PydanticOutputParser` 将 LLM 的非结构化文本输出强制转换为 Pydantic 结构化数据模型。通过 LangChain Expression Language（LCEL）的链式调用范式，系统将"检索→拼接 Prompt→调用 LLM→解析输出"这一完整流水线组织为清晰、可维护的代码结构。

**向量数据库层：ChromaDB。** ChromaDB 是一个轻量级的开源向量数据库，专为 LLM 应用的 RAG 场景设计。本系统选择 ChromaDB 的核心原因在于其"零配置"特性：只需 `pip install chromadb` 即可使用，无需安装 Docker、无需启动独立服务进程、无需配置网络端口。ChromaDB 以本地文件系统目录（`./chroma_db/`）作为持久化存储，支持余弦相似度（Cosine Similarity）语义检索，并提供 `similarity_search` 等高层 API，与 LangChain 的集成极为顺畅。在本系统中，PDF 教材经嵌入模型向量化后存入 ChromaDB，后续出题和批改时通过语义检索获取最相关的 Top-K 文本片段作为 LLM 的 Prompt 上下文。

**关系型数据库层：SQLite。** SQLite 是 Python 标准库内置的嵌入式关系型数据库，无需任何安装或配置步骤。本系统使用 SQLite 存储五类结构化数据：用户账户（users 表，含 SHA-256 密码哈希与角色字段）、教师发布的习题（exercises 表，含 JSON 格式的标准答案要点）、学生答题记录（exam_records 表，含 CoT 各步得分与薄弱知识点标签）、知识点错误统计（knowledge_tags 表，用于聚合薄弱点分布图）以及聊天消息（messages 表，含文件路径与文件类型元数据）。所有数据库操作均通过参数化 SQL 查询（使用 `?` 占位符）实现，杜绝 SQL 注入风险。

**嵌入模型层：all-MiniLM-L6-v2（HuggingFace）。** 在向量嵌入环节，本系统采用 `sentence-transformers/all-MiniLM-L6-v2` 模型。这是一个基于 Microsoft MiniLM 架构的轻量级 Sentence Transformer，输出 384 维稠密向量。选择该模型的最关键理由是：它完全在本地 CPU 上运行，无需调用任何云端 Embedding API，因此不产生任何费用且不受网络延迟影响。模型通过 `HuggingFaceEmbeddings` 类（来自 `langchain-huggingface` 包）加载，首次运行时自动从 HuggingFace Hub 下载模型权重（约 80 MB），之后离线运行。该模型以英文为主，但对中英文混合文本也具有良好的泛化能力，能够满足本系统处理中英双语教学材料的检索需求。

### Русский текст

При выборе технологий для данной системы строго соблюдались принципы «дружественности к начинающим, низкой стоимости и быстрой поставки», исключающие внедрение компонентов, требующих сложной эксплуатации или высоких аппаратных затрат. Вся система построена на экосистеме Python — от фронтенда до бэкенда, от персистентности данных до AI-вывода — с использованием зрелых open-source решений с активным сообществом и качественной документацией. Ниже приводится послойное описание каждого технологического выбора и его обоснование.

**Фронтенд-слой: Streamlit 1.57.** Streamlit — это чисто Python-фреймворк для веб-приложений, специально разработанный для задач data science и машинного обучения. Разработчику не требуется писать ни одной строки на HTML, CSS или JavaScript — достаточно вызовов функций Python для построения веб-приложений с формами, графиками, загрузкой файлов, чат-интерфейсами и другим богатым интерактивным функционалом. В условиях单人-разработки данного проекта Streamlit сводит затраты на фронтенд-разработку практически к нулю, позволяя разработчику полностью сосредоточиться на реализации核心 AI-логики. Кроме того, Streamlit изначально поддерживает декоратор кэширования ресурсов `st.cache_resource`, управление состоянием сессии `st.session_state`, частичное обновление страниц `st.fragment` и механизм автоматического обновления по таймеру `@st.fragment(run_every=N)`. Эти возможности обеспечивают элегантную и эффективную реализацию «квази-реального» обновления сообщений в чат-системе и сохранения состояния между страницами.

**Слой API больших языковых моделей: DeepSeek-V3.** DeepSeek-V3 — это новейшая большая языковая модель, выпущенная компанией 深度求索 (DeepSeek), достигающая международного передового уровня по таким ключевым измерениям, как способность к рассуждению, математические способности и навыки программирования. Выбор DeepSeek в качестве базовой модели для данной системы основан на трёх соображениях. Во-первых, ценообразование API DeepSeek чрезвычайно конкурентоспособно (около 1 юаня за миллион токенов), что крайне выгодно для частой отладки и масштабного тестирования на стадии дипломного проектирования. Во-вторых, API DeepSeek полностью совместим со спецификацией интерфейса OpenAI SDK, что позволяет напрямую использовать пакет `langchain-openai` для интеграции без дополнительных адаптационных работ. В-третьих, DeepSeek-V3 демонстрирует превосходную производительность как на китайском, так и на английском языках, хорошо поддерживая двуязычный образовательный сценарий данной системы. Система вызывает API через класс `ChatOpenAI` (совместимый интерфейс) с низким значением параметра temperature=0.1 для запроса Chat Completion, обеспечивая согласованность и воспроизводимость оценок.

**Слой оркестрации: LangChain + LCEL.** LangChain является наиболее распространённым стандартным фреймворком в области разработки LLM-приложений, предоставляя ряд готовых к использованию компонентов: загрузчики документов (Document Loader), разделители текста (Text Splitter), векторные хранилища (Vector Store), парсеры вывода (Output Parser) и другие. В данной системе используются следующие возможности LangChain: `PyPDFLoader` для загрузки PDF-материалов, `RecursiveCharacterTextSplitter` для сегментации текста по семантическим границам, `Chroma` для инкапсуляции интерфейсов хранения и поиска ChromaDB, `PydanticOutputParser` для принудительного преобразования неструктурированного текстового вывода LLM в структурированные модели данных Pydantic. С помощью парадигмы цепочечных вызовов LangChain Expression Language (LCEL) система организует полный конвейер «поиск → формирование промпта → вызов LLM → разбор вывода» в ясную и удобную для сопровождения структуру кода.

**Слой векторной базы данных: ChromaDB.** ChromaDB — это легковесная открытая векторная база данных, специально спроектированная для сценариев RAG в LLM-приложениях. Ключевая причина выбора ChromaDB для данной системы — её характеристика «нулевой конфигурации»: достаточно выполнить `pip install chromadb`, не требуется установка Docker, запуск отдельных сервисных процессов или настройка сетевых портов. ChromaDB использует директорию локальной файловой системы (`./chroma_db/`) в качестве персистентного хранилища, поддерживает семантический поиск по косинусному сходству (Cosine Similarity) и предоставляет высокоуровневые API, такие как `similarity_search`, с чрезвычайно гладкой интеграцией с LangChain. В данной системе PDF-материалы после векторизации модели эмбеддинга сохраняются в ChromaDB; при последующей генерации заданий и оценивании через семантический поиск извлекаются наиболее релевантные Top-K текстовые фрагменты в качестве контекста промпта для LLM.

**Слой реляционной базы данных: SQLite.** SQLite — это встроенная реляционная база данных, входящая в стандартную библиотеку Python и не требующая каких-либо шагов по установке или настройке. В данной системе SQLite используется для хранения пяти типов структурированных данных: учётные записи пользователей (таблица users, содержит SHA-256 хеш пароля и поле роли), задания, опубликованные преподавателем (таблица exercises, содержит критерии правильного ответа в формате JSON), записи ответов студентов (таблица exam_records, содержит баллы каждого шага CoT и метки слабых мест), статистика ошибок по темам (таблица knowledge_tags, используется для агрегации распределения слабых мест) и сообщения чата (таблица messages, содержит путь к файлу и метаданные о типе файла). Все операции с базой данных выполняются через параметризованные SQL-запросы (с использованием плейсхолдеров `?`), что полностью исключает риск SQL-инъекций.

**Слой модели эмбеддинга: all-MiniLM-L6-v2 (HuggingFace).** В части векторного эмбеддинга система использует модель `sentence-transformers/all-MiniLM-L6-v2`. Это легковесный Sentence Transformer на основе архитектуры Microsoft MiniLM, выдающий плотные векторы размерности 384. Ключевое преимущество выбора данной модели заключается в том, что она полностью работает на локальном CPU, не требуя вызова каких-либо облачных Embedding API, и, следовательно, не создаёт никаких расходов и не подвержена влиянию сетевых задержек. Модель загружается через класс `HuggingFaceEmbeddings` (из пакета `langchain-huggingface`), при первом запуске автоматически скачивает веса модели (около 80 МБ) с HuggingFace Hub, после чего работает офлайн. Модель ориентирована преимущественно на английский язык, но также обладает хорошей способностью к обобщению на смешанных китайско-английских текстах, что позволяет удовлетворить потребности поиска в двуязычных учебных материалах данной системы.

### 2.1.1 目录结构与模块职责 / Структура каталогов и обязанности модулей

### 中文正文

本系统的全部源代码、配置文件和运行时数据均组织于 `НИРС/` 项目根目录下。该目录采用扁平化的模块组织方式——每个 `.py` 文件对应一个功能独立的代码模块，无深层嵌套的子包结构，便于单人开发者快速定位和维护代码。以下从代码文件、配置与数据两个维度对目录结构进行说明。

**代码文件（5 个核心模块）：**

`main.py`（726 行）是整个系统的入口文件，承担三项核心职责。第一，登录与注册页面的渲染与逻辑处理——包括角色选择（教师/学生）、表单验证、密码哈希比对、Session State 初始化。第二，全局侧边栏导航的渲染——根据当前用户的角色（教师或学生）动态生成不同的菜单项，并处理页面切换时的状态清理。第三，全部 8 个功能页面的 UI 渲染与交互逻辑——包括教师端的教材管理、习题生成、学生成绩、师生聊天四个页面，以及学生端的习题作答、我的成绩、错题集、师生聊天四个页面。由于 Streamlit 框架采用"脚本即页面"的执行模型，`main.py` 在每次用户交互时从头至尾重新执行，因此所有页面的逻辑均集中于此文件中。

`rag_engine.py`（102 行）封装了 RAG 检索增强生成的核心能力。该模块定义了 `RAGEngine` 类，内部持有三个关键组件：HuggingFace 嵌入模型（`all-MiniLM-L6-v2`）、递归字符文本分割器（`RecursiveCharacterTextSplitter`）、以及 ChromaDB 向量存储实例。模块对上层暴露四个简洁的公共方法：`load_pdf()` 完成"PDF 加载→文本切分→向量化→持久化"的完整流水线；`search()` 执行语义相似度检索并返回纯文本片段列表；`search_with_sources()` 额外返回来源元数据（文件名、页码）；`get_chunks_count()` 查询当前知识库规模。

`cot_grader.py`（136 行）封装了基于思维链的智能出题与批改能力。该模块定义了 `CoTGrader` 类，内部持有 DeepSeek Chat API 客户端实例和 Pydantic 输出解析器。模块还定义了四组 Pydantic 数据模型（`AnswerPoint`、`QuestionOutput`、`StepScore`、`GradingOutput`），用于强制约束 LLM 输出的 JSON 结构。两个核心方法 `generate_question()` 和 `grade_answer()` 均采用"Prompt 注入→调用 LLM→提取 JSON→失败重试"的容错流水线，确保系统在 LLM 输出格式异常时能够自动恢复。

`prompts.py`（94 行）集中管理了系统中全部 LLM 提示词模板。该模块定义了三个核心 Prompt 常量：`QUESTION_GENERATION_PROMPT`（出题模板，约 30 行）、`COT_GRADING_PROMPT`（CoT 批改模板，约 40 行）和 `WEAKNESS_ANALYSIS_PROMPT`（薄弱点分析模板，约 15 行）。每个 Prompt 均包含角色设定、输入变量占位符、详细的步骤指令和严格的 JSON 输出格式示例。将 Prompt 与业务逻辑分离的模块化设计，使得 Prompt 的调试、优化和多语言适配可以独立进行，无需修改 `cot_grader.py` 中的业务代码。

`db.py`（358 行）是系统的数据持久化层，封装了所有与 SQLite 数据库的交互操作。该模块定义了 6 张数据表的 DDL（`users`、`exercises`、`exam_records`、`knowledge_tags`、`messages`），并提供了覆盖用户管理、习题管理、答题记录、统计分析和聊天消息五大功能域的 20 余个函数。所有 SQL 查询均使用参数化占位符（`?`），密码通过 SHA-256 哈希存储。

**配置与数据目录：**

`.streamlit/config.toml` 包含 Streamlit 框架的运行配置，当前设置 `toolbarMode = "viewer"` 以隐藏右上角的部署工具栏，保持教学场景界面的简洁性。

`.env` 文件存储敏感配置信息（DeepSeek API Key 和 Base URL），通过 `python-dotenv` 库加载为环境变量，与代码分离以避免密钥泄露。

`data/` 目录用于存放教师上传的 PDF 教材原始文件，支持通过 `st.file_uploader` 实时上传或直接放入目录后通过下拉菜单快速加载。

`chroma_db/` 目录是 ChromaDB 的持久化向量存储路径。PDF 教材经文本切分和嵌入向量化后，以 ChromaDB 的内部格式存储于此。该目录的存在使得系统重启后无需重新处理已加载的教材。

`uploads/chat/` 目录用于存储师生聊天中传输的文件。文件以"时间戳_原始文件名"的格式重命名保存，避免重名冲突。

### Русский текст

Весь исходный код, конфигурационные файлы и данные времени выполнения данной системы организованы в корневом каталоге проекта `НИРС/`. Каталог использует плоскую модульную организацию — каждый `.py` файл соответствует функционально независимому кодовому модулю без глубоко вложенных подпакетов, что позволяет одному разработчику быстро находить и сопровождать код. Ниже приводится описание структуры каталогов в двух измерениях: кодовые файлы и конфигурация/данные.

**Кодовые файлы (5 основных модулей):**

`main.py` (726 строк) является входным файлом всей системы и выполняет три ключевые обязанности. Во-первых, рендеринг и логика страницы входа/регистрации — включая выбор роли (преподаватель/студент), валидацию форм, сравнение хешей паролей, инициализацию Session State. Во-вторых, рендеринг глобальной боковой панели навигации — динамическая генерация различных пунктов меню в зависимости от роли текущего пользователя (преподаватель или студент) и обработка очистки состояния при переключении страниц. В-третьих, рендеринг UI и логика взаимодействия всех 8 функциональных страниц — включая четыре страницы преподавателя (управление материалами, генерация заданий, оценки студентов, чат) и четыре страницы студента (ответ на задания, мои оценки, сборник ошибок, чат). `rag_engine.py` (102 строки) инкапсулирует ключевые возможности поисково-дополненной генерации RAG. Данный модуль определяет класс `RAGEngine`, который внутренне владеет тремя ключевыми компонентами: моделью эмбеддинга HuggingFace (`all-MiniLM-L6-v2`), рекурсивным символьным разделителем текста (`RecursiveCharacterTextSplitter`) и экземпляром векторного хранилища ChromaDB. Модуль предоставляет верхнему уровню четыре лаконичных публичных метода: `load_pdf()` выполняет полный конвейер «загрузка PDF → сегментация текста → векторизация → персистентность»; `search()` выполняет семантический поиск по сходству и возвращает список текстовых фрагментов; `search_with_sources()` дополнительно возвращает метаданные источника (имя файла, номер страницы); `get_chunks_count()` запрашивает текущий размер базы знаний.

`cot_grader.py` (136 строк) инкапсулирует возможности интеллектуальной генерации заданий и оценивания на основе цепочки рассуждений. Данный модуль определяет класс `CoTGrader`, который внутренне владеет экземпляром клиента DeepSeek Chat API и парсерами вывода Pydantic. Модуль также определяет четыре группы моделей данных Pydantic (`AnswerPoint`, `QuestionOutput`, `StepScore`, `GradingOutput`) для принудительного ограничения JSON-структуры вывода LLM. Два ключевых метода `generate_question()` и `grade_answer()` используют отказоустойчивый конвейер «вставка промпта → вызов LLM → извлечение JSON → повтор при ошибке», обеспечивая автоматическое восстановление системы при аномальном формате вывода LLM.

`prompts.py` (94 строки) централизованно управляет всеми шаблонами промптов LLM в системе. Данный модуль определяет три основные константы промптов: `QUESTION_GENERATION_PROMPT` (шаблон генерации заданий, около 30 строк), `COT_GRADING_PROMPT` (шаблон оценивания CoT, около 40 строк) и `WEAKNESS_ANALYSIS_PROMPT` (шаблон анализа слабых мест, около 15 строк). Каждый промпт содержит определение роли, плейсхолдеры входных переменных, подробные пошаговые инструкции и строгие примеры вывода в формате JSON. Модульная конструкция, отделяющая промпты от бизнес-логики, позволяет независимо отлаживать, оптимизировать и адаптировать промпты под разные языки без изменения кода в `cot_grader.py`.

`db.py` (358 строк) является слоем персистентности данных системы, инкапсулируя все операции взаимодействия с базой данных SQLite. Данный модуль определяет DDL шести таблиц данных (`users`, `exercises`, `exam_records`, `knowledge_tags`, `messages`) и предоставляет более 20 функций, охватывающих пять функциональных областей: управление пользователями, управление заданиями, записи ответов, статистический анализ и сообщения чата. Все SQL-запросы используют параметризованные плейсхолдеры (`?`), пароли хранятся в виде SHA-256 хешей.

**Конфигурация и каталоги данных:**

`.streamlit/config.toml` содержит конфигурацию выполнения фреймворка Streamlit с текущей настройкой `toolbarMode = "viewer"` для скрытия панели развёртывания в правом верхнем углу и сохранения лаконичного интерфейса образовательного сценария.

Файл `.env` хранит конфиденциальную информацию конфигурации (DeepSeek API Key и Base URL), загружаемую в переменные окружения через библиотеку `python-dotenv` и отделённую от кода для предотвращения утечки ключей.

Каталог `data/` используется для хранения исходных PDF-файлов учебных материалов, загруженных преподавателем. Поддерживается как загрузка в реальном времени через `st.file_uploader`, так и быстрое подключение через выпадающее меню после размещения файлов в каталоге.

Каталог `chroma_db/` является путём персистентного векторного хранилища ChromaDB. PDF-материалы после сегментации текста и векторизации эмбеддинга сохраняются здесь во внутреннем формате ChromaDB. Наличие этого каталога позволяет системе не переобрабатывать уже загруженные материалы после перезапуска.

Каталог `uploads/chat/` используется для хранения файлов, передаваемых в чате между преподавателем и студентом. Файлы сохраняются с переименованием в формате «временная_метка_оригинальное_имя_файла» для предотвращения конфликтов имён.

### 2.1.2 模块依赖关系 / Зависимости модулей

### 中文正文

本系统的五个核心模块之间形成了清晰的层次化依赖关系，整体呈现出以 `main.py` 为顶层调度中心、`rag_engine.py` 和 `cot_grader.py` 为中层能力引擎、`prompts.py` 和 `db.py` 为底层支撑模块的三层架构。

**依赖链条详细说明：**

`main.py` 作为系统入口，依赖于其余全部四个模块。它从 `rag_engine` 导入 `RAGEngine` 类，通过 `@st.cache_resource` 装饰器将其实例化为全局单例，供教材管理和习题生成页面调用；从 `cot_grader` 导入 `CoTGrader` 类，同样以缓存单例模式实例化，供出题和批改流程使用；从 `db` 导入全部数据库操作函数（约 15 个），涵盖用户认证、习题存取、答题记录、统计分析和聊天消息等全部数据操作；同时作为 Streamlit 应用，`main.py` 完整依赖 Streamlit 框架提供的 UI 组件（`st.button`、`st.text_area`、`st.chat_message`、`st.file_uploader`、`st.plotly_chart` 等）、会话管理（`st.session_state`）和页面刷新机制（`st.rerun`、`@st.fragment`）。

`rag_engine.py` 的依赖链包含三个外部技术组件。在文档加载层，它依赖 LangChain 的 `PyPDFLoader` 解析 PDF 文件；在文本处理层，依赖 `RecursiveCharacterTextSplitter` 按语义边界切分长文本；在向量化与检索层，依赖 `HuggingFaceEmbeddings`（封装自 `langchain-huggingface` 包）调用本地的 `all-MiniLM-L6-v2`  Sentence Transformer 模型生成文本嵌入向量，并依赖 `Chroma`（封装自 `langchain-community` 包）进行向量的持久化存储与相似度检索。

`cot_grader.py` 的依赖链分为三个方向。在 LLM 调用方向，它依赖 `ChatOpenAI`（来自 `langchain-openai` 包），该客户端以 OpenAI 兼容模式连接至 DeepSeek API 端点，发送 Chat Completion 请求并接收 LLM 生成的文本；在 Prompt 方向，它从 `prompts.py` 导入 `QUESTION_GENERATION_PROMPT` 和 `COT_GRADING_PROMPT` 两个模板常量，通过 Python 字符串的 `.format()` 方法将变量注入模板；在输出解析方向，它依赖 `PydanticOutputParser`（来自 `langchain-core` 包）和自定义的四组 Pydantic 数据模型，将 LLM 的非结构化 JSON 文本强制校验为类型安全的 Python 字典。

`prompts.py` 是本系统中唯一无任何外部依赖的纯数据模块。它仅定义字符串常量，不导入任何项目内模块或第三方包。

`db.py` 仅依赖 Python 标准库中的 `sqlite3` 模块和 `hashlib` 模块（用于 SHA-256 密码哈希），以及标准库的 `json` 模块（用于序列化/反序列化答题记录中的结构化字段）。该模块不依赖任何第三方包，也不依赖项目内的其他模块。这种"零外部依赖"的设计使得数据库层可以在不启动 Streamlit、不加载 AI 模型的情况下独立进行单元测试。

### Русский текст

Пять основных модулей данной системы образуют чёткую иерархическую структуру зависимостей, в целом представляющую трёхуровневую архитектуру, где `main.py` выступает в роли верхнего диспетчерского центра, `rag_engine.py` и `cot_grader.py` — среднего уровня движков возможностей, а `prompts.py` и `db.py` — нижнего уровня поддерживающих модулей.

**Детальное описание цепочек зависимостей:**

`main.py`, как входная точка системы, зависит от всех остальных четырёх модулей. Из `rag_engine` он импортирует класс `RAGEngine`, который с помощью декоратора `@st.cache_resource` инстанцируется как глобальный синглтон для использования на страницах управления материалами и генерации заданий. Из `cot_grader` он импортирует класс `CoTGrader`, также инстанцируемый в режиме кэшированного синглтона для процессов генерации заданий и оценивания. Из `db` он импортирует все функции операций с базой данных (около 15), охватывая аутентификацию пользователей, доступ к заданиям, записи ответов, статистический анализ и сообщения чата. Одновременно, как приложение Streamlit, `main.py` полностью зависит от UI-компонентов фреймворка Streamlit (`st.button`, `st.text_area`, `st.chat_message`, `st.file_uploader`, `st.plotly_chart` и др.), управления сессиями (`st.session_state`) и механизмов обновления страниц (`st.rerun`, `@st.fragment`).

Цепочка зависимостей `rag_engine.py` включает три внешних технических компонента. На уровне загрузки документов он зависит от `PyPDFLoader` из LangChain для парсинга PDF-файлов. На уровне обработки текста — от `RecursiveCharacterTextSplitter` для сегментации длинных текстов по семантическим границам. На уровне векторизации и поиска — от `HuggingFaceEmbeddings` (инкапсулированного из пакета `langchain-huggingface`) для вызова локальной модели Sentence Transformer `all-MiniLM-L6-v2` и генерации эмбеддингов текста, и от `Chroma` (инкапсулированного из пакета `langchain-community`) для персистентного хранения векторов и поиска по сходству.

Цепочка зависимостей `cot_grader.py` разделяется на три направления. В направлении вызова LLM он зависит от `ChatOpenAI` (из пакета `langchain-openai`) — клиента, который подключается к конечной точке DeepSeek API в режиме совместимости с OpenAI, отправляет запросы Chat Completion и получает сгенерированный LLM текст. В направлении промптов он импортирует из `prompts.py` две шаблонные константы — `QUESTION_GENERATION_PROMPT` и `COT_GRADING_PROMPT` — и вставляет переменные в шаблоны с помощью метода `.format()` строк Python. В направлении разбора вывода он зависит от `PydanticOutputParser` (из пакета `langchain-core`) и четырёх пользовательских моделей данных Pydantic, принудительно проверяющих неструктурированный JSON-текст LLM и преобразующих его в типобезопасные словари Python.

`prompts.py` — единственный чисто данных модуль в системе, не имеющий никаких внешних зависимостей. Он определяет только строковые константы и не импортирует ни внутрипроектные модули, ни сторонние пакеты.

`db.py` зависит только от модуля `sqlite3` и модуля `hashlib` из стандартной библиотеки Python (для SHA-256 хеширования паролей), а также от модуля `json` (для сериализации/десериализации структурированных полей в записях ответов). Данный модуль не зависит ни от каких сторонних пакетов, ни от других модулей внутри проекта. Такая конструкция «нулевых внешних зависимостей» позволяет независимо выполнять модульное тестирование слоя базы данных без запуска Streamlit и загрузки AI-моделей.

В следующей таблице обобщены внешние зависимости каждого модуля:

| Модуль | Внутрипроектные зависимости | Сторонние пакеты | Внешние сервисы |
|--------|---------------------------|------------------|-----------------|
| main.py | rag_engine, cot_grader, db | streamlit, plotly, pandas | — |
| rag_engine.py | — | langchain-community, langchain-text-splitters, langchain-huggingface, chromadb, sentence-transformers | HuggingFace Hub (первая загрузка модели) |
| cot_grader.py | prompts | langchain-openai, langchain-core, pydantic | DeepSeek API |
| prompts.py | — | — | — |
| db.py | — | — (только стандартная библиотека Python) | — |

## 3.1 RAG 检索增强生成流水线 / Конвейер RAG

本节围绕 RAG 技术的两个核心环节展开：3.1.1 详解从 PDF 上传到向量检索的完整文档处理流水线（四个阶段的技术组件与输入输出）；3.1.2 阐述系统从知识锚定、检索量配置到 Prompt 约束的三层防幻觉策略。

Данный раздел рассматривает два ключевых аспекта технологии RAG: в разделе 3.1.1 детально описывается полный конвейер обработки документов от загрузки PDF до векторного поиска (четыре этапа с указанием компонентов и входных/выходных данных); в разделе 3.1.2 излагается трёхуровневая стратегия предотвращения галлюцинаций — привязка к знаниям, настройка объёма поиска и ограничения на уровне промпта.

### 3.1.1 文档处理流程 / Процесс обработки документов

### 中文正文

RAG 流水线的起点是教师上传的 PDF 教材文件。`rag_engine.py` 中的 `load_pdf()` 方法封装了从 PDF 到向量检索的完整流水线，依次执行四个阶段：

**第一阶段：PDF 解析。** 系统使用 LangChain 的 `PyPDFLoader` 加载 PDF 文件。`PyPDFLoader` 基于 `pypdf` 库将 PDF 中的每一页提取为一个独立的 `Document` 对象，保留页面内容与元数据（页码、来源文件名）。该阶段输出为 `List[Document]`，每个元素对应一页文本。

**第二阶段：文本切分。** 解析后的长文本无法直接用于向量化检索（单页文档可能包含数千字），需要按语义边界切分为较小的文本块。系统使用 `RecursiveCharacterTextSplitter`，按优先级依次尝试 `["\n\n", "\n", "。", ".", " ", ""]` 作为分隔符，确保切分位置尽量保持在段落、句子等自然边界处。配置参数为 chunk_size=500（每块最多 500 字符）、chunk_overlap=50（相邻块之间重叠 50 字符以避免语义断裂）。该阶段输出为 `List[Document]`，每个元素长度约为 500 字符。

**第三阶段：向量化嵌入。** 每个文本块被传入 `HuggingFaceEmbeddings`，经过 `all-MiniLM-L6-v2` Sentence Transformer 模型计算为 384 维的稠密浮点向量。该模型完全在本地 CPU 上运行，无需网络请求，无 API 费用。模型输出向量经过 L2 归一化，使得后续的余弦相似度检索等价于内积运算。

**第四阶段：向量存储与索引。** 生成的向量与对应的文本块、元数据一同存入 ChromaDB。ChromaDB 在本地 `./chroma_db/` 目录中以 HNSW（Hierarchical Navigable Small World）算法构建近似最近邻索引，支持高效的语义相似度搜索。后续检索时，用户查询同样经过相同的嵌入模型向量化，然后通过 `similarity_search(k=5)` 返回与查询最相似的 Top-5 文本片段。

下表概括了每个阶段的输入、输出与使用的技术组件：

| 阶段 / Этап | 输入 / Вход | 输出 / Выход | 使用组件 / Компонент |
|------------|------------|-------------|-------------------|
| PDF 解析 | PDF 文件路径 | `List[Document]`（每页一个） | `PyPDFLoader` (langchain-community) |
| 文本切分 | `List[Document]`（整页） | `List[Document]`（500字符/块） | `RecursiveCharacterTextSplitter` |
| 向量化嵌入 | 文本块 | `List[float]`（384维向量） | `HuggingFaceEmbeddings` (all-MiniLM-L6-v2) |
| 向量存储 | 向量 + 文本 + 元数据 | ChromaDB 持久化索引 | `Chroma` (langchain-community) |

### Русский текст

Отправной точкой конвейера RAG является PDF-файл учебных материалов, загруженный преподавателем. Метод `load_pdf()` в `rag_engine.py` инкапсулирует полный конвейер от PDF до векторного поиска, последовательно выполняя четыре этапа.

**Первый этап: парсинг PDF.** Система использует `PyPDFLoader` из LangChain для загрузки PDF-файла. `PyPDFLoader` на основе библиотеки `pypdf` извлекает каждую страницу PDF в отдельный объект `Document`, сохраняя содержимое страницы и метаданные (номер страницы, имя файла источника). Результатом этапа является `List[Document]`, где каждый элемент соответствует одной странице текста.

**Второй этап: сегментация текста.** Длинный текст после парсинга непригоден для прямой векторизации и поиска (одна страница может содержать тысячи символов) — его необходимо разделить на небольшие фрагменты по семантическим границам. Система использует `RecursiveCharacterTextSplitter`, последовательно перебирая разделители `["\n\n", "\n", "。", ".", " ", ""]` в порядке приоритета, чтобы обеспечить разбивку на естественных границах (абзацы, предложения). Параметры: chunk_size=500 (максимум 500 символов на фрагмент), chunk_overlap=50 (перекрытие 50 символов между соседними фрагментами для предотвращения потери семантики). Результатом этапа является `List[Document]` с длиной каждого элемента около 500 символов.

**Третий этап: векторное эмбеддинг.** Каждый текстовый фрагмент передаётся в `HuggingFaceEmbeddings`, который вычисляет плотный вектор размерности 384 с помощью Sentence Transformer модели `all-MiniLM-L6-v2`. Модель полностью работает на локальном CPU без сетевых запросов и без затрат на API. Выходной вектор нормализуется по L2, что делает последующее вычисление косинусного сходства эквивалентным скалярному произведению.

**Четвёртый этап: хранение векторов и индексация.** Сгенерированные векторы вместе с соответствующими текстовыми фрагментами и метаданными сохраняются в ChromaDB. ChromaDB строит индекс приближённого ближайшего соседа на основе алгоритма HNSW (Hierarchical Navigable Small World) в локальной директории `./chroma_db/`, обеспечивая эффективный семантический поиск. При последующем поиске запрос пользователя векторизуется той же моделью эмбеддинга, после чего `similarity_search(k=5)` возвращает Top-5 наиболее релевантных текстовых фрагментов.

В следующей таблице обобщены входные данные, выходные данные и используемые компоненты каждого этапа:

| Этап | Вход | Выход | Компонент |
|------|------|-------|-----------|
| Парсинг PDF | Путь к PDF-файлу | `List[Document]` (постранично) | `PyPDFLoader` (langchain-community) |
| Сегментация текста | `List[Document]` (целые страницы) | `List[Document]` (500 символов/фрагмент) | `RecursiveCharacterTextSplitter` |
| Векторный эмбеддинг | Текстовый фрагмент | `List[float]` (384-мерный вектор) | `HuggingFaceEmbeddings` (all-MiniLM-L6-v2) |
| Хранение векторов | Вектор + текст + метаданные | Персистентный индекс ChromaDB | `Chroma` (langchain-community) |

### 3.1.2 防幻觉策略 / Стратегия предотвращения галлюцинаций

### 中文正文

大语言模型在教育场景中最致命的问题是"幻觉"——模型可能生成与教材内容相悖的题目和批改意见。本系统从三个层面构建防幻觉机制：

**第一，知识锚定（Knowledge Anchoring）。** 系统在出题和批改的每个 LLM 调用中，均通过 RAG 检索将与当前任务最相关的教材片段注入 Prompt。注入的文本是教材原始内容，而非模型的内部参数知识。这种"外部知识锚定"机制从根本上确保 LLM 的输出不会偏离教材内容。

**第二，检索量配置。** 系统设定检索数量 k=5，即每次调用获取 5 个最相关的文本片段（约 2500 字符上下文）。这一数量在"足够覆盖题目所需知识点范围"和"不超过 LLM 上下文窗口限制"之间取得平衡。过少的检索量可能导致上下文不足，过多的检索量可能引入噪声或稀释注意力。

**第三，Prompt 级约束。** 系统在 `QUESTION_GENERATION_PROMPT` 和 `COT_GRADING_PROMPT` 中均以显式指令约束 LLM 的行为边界，包括：（1）"严格基于上述教材内容生成题目，不得自行编造知识点"；（2）"如果学生答案超出教材范围，在反馈中说明但不要扣分"；（3）"所有引用必须可在教材中找到对应原文"。这些指令以自然语言嵌入 Prompt，通过 InstructGPT 的指令跟随能力强制执行。

下表总结了三层防幻觉策略及其作用范围：

| 策略层级 / Уровень стратегии | 具体措施 / Мера | 作用阶段 / Этап | 防护目标 / Цель |
|---------------------------|---------------|----------------|----------------|
| 知识锚定 | RAG 检索注入教材原文 | 出题 + 批改 | 从根本上防止知识偏移 |
| 检索量配置 | k=5, 约 2500 字符上下文 | 出题 + 批改 | 保证充分上下文覆盖 |
| Prompt 约束 | 显式指令限制行为边界 | 出题 + 批改 | 约束 LLM 行为 |

### Русский текст

Наиболее критическая проблема больших языковых моделей в образовательных сценариях — это «галлюцинации»: модель может генерировать задания и оценочные комментарии, противоречащие содержанию учебных материалов. Данная система выстраивает механизм предотвращения галлюцинаций на трёх уровнях.

**Первый уровень: привязка к знаниям (Knowledge Anchoring).** При каждом вызове LLM для генерации заданий или оценивания система через RAG извлекает наиболее релевантные фрагменты учебных материалов и встраивает их в промпт. Внедряемый текст является оригинальным содержимым учебных материалов, а не внутренними параметрическими знаниями модели. Этот механизм «внешней привязки к знаниям» принципиально гарантирует, что вывод LLM не отклонится от содержания учебных материалов.

**Второй уровень: настройка объёма поиска.** Система устанавливает количество извлекаемых фрагментов k=5, то есть при каждом вызове получает 5 наиболее релевантных текстовых фрагментов (около 2500 символов контекста). Это значение обеспечивает баланс между «достаточным покрытием объёма знаний, необходимого для задания» и «ограничением контекстного окна LLM». Слишком малое количество фрагментов может привести к недостаточности контекста, слишком большое — внести шум или ослабить внимание модели.

**Третий уровень: ограничения на уровне промпта.** Система включает в `QUESTION_GENERATION_PROMPT` и `COT_GRADING_PROMPT` явные инструкции, ограничивающие поведение LLM: (1) «строго основываться на вышеуказанном содержании учебных материалов, не выдумывать знания»; (2) «если ответ студента выходит за рамки материалов, указать это в обратной связи, но не снижать оценку»; (3) «все ссылки должны иметь соответствие в исходном тексте учебных материалов». Эти инструкции встраиваются в промпт на естественном языке и обеспечиваются через способность модели следовать инструкциям.

В следующей таблице обобщены три уровня стратегии предотвращения галлюцинаций и их область действия:

| Уровень стратегии | Мера | Этап применения | Цель защиты |
|------------------|------|-----------------|-------------|
| Привязка к знаниям | Внедрение оригинального текста через RAG | Генерация + Оценивание | Принципиальное предотвращение отклонения знаний |
| Настройка объёма поиска | k=5, около 2500 символов контекста | Генерация + Оценивание | Обеспечение достаточного контекстного покрытия |
| Ограничения промпта | Явные инструкции, ограничивающие поведение | Генерация + Оценивание | Контроль поведения LLM |

## 3.2 CoT 思维链批改机制 / Механизм оценивания CoT

本节聚焦 CoT 批改引擎的内部机制：3.2.1 剖析"要点匹配→逻辑评估→综合反馈"三步推理链的设计逻辑与各步输出结构；3.2.2 说明少样本提示与 PydanticOutputParser 如何协同保障 LLM 输出格式的一致性和稳定性。

Данный раздел фокусируется на внутреннем механизме движка оценивания CoT: в разделе 3.2.1 анализируется логика трёхшаговой цепочки рассуждений «сопоставление критериев → оценка логики → комплексная обратная связь» и структура выходных данных каждого шага; в разделе 3.2.2 описывается, как Few-Shot Prompting совместно с PydanticOutputParser обеспечивают согласованность и стабильность формата вывода LLM.

### 3.2.1 三步推理链 / Трёхшаговая цепочка рассуждений

### 中文正文

本系统的核心批改逻辑由 `cot_grader.py` 中的 `grade_answer()` 方法实现。该方法采用思维链提示工程（Chain of Thought Prompting），强制 LLM 按照三个推理步骤依次执行批改，每一步输出结构化的中间结果，最终汇总为完整的评价。这种分步设计使批改过程透明、可解释、可复核。

**步骤一：要点逐条匹配（Point-wise Matching）。** 系统首先将标准答案中的每个得分点（`standard_answer_points`）与学生答案逐条进行语义匹配。对于每个得分点，LLM 需要判断学生是否涵盖了该要点，涵盖程度如何，并给出 0-5 分的具体评分和评语。此步骤的输出是 `step_scores` 数组，每个元素包含 `point_index`（得分点索引）、`student_content`（学生相关内容摘录）、`score`（0-5 分）和 `comment`（评分理由）。最大总分由得分点数量乘以 5 计算得出。

**步骤二：整体逻辑评估（Logic Assessment）。** 在学生答案的要点覆盖之外，LLM 需要对答案的整体逻辑进行独立评估。评估维度包括：推理链条是否完整、论证是否自洽、是否存在逻辑跳跃或矛盾。此步骤输出单一的逻辑分（`logic_score`，0-5 分）和对应的评语（`logic_comment`）。逻辑分的加入弥补了"逐点匹配"模式无法评估整体论证质量的缺陷。

**步骤三：综合反馈与薄弱点标注（Comprehensive Feedback）。** 综合前两步的评分结果，LLM 计算总分（`total_score = sum(step_scores.score) + logic_score`），并生成三段式反馈：综合评价短语、苏格拉底式引导提问、薄弱知识点标签（`weak_tags`）。这些标签将写入 `knowledge_tags` 表，用于后续的薄弱知识点聚合分析。

下表总结了三步推理链的输入、输出和评估维度：

| 步骤 / Шаг | 名称 / Название | 输入 / Вход | 输出 / Выход | 评估维度 / Измерение |
|-----------|----------------|------------|-------------|-------------------|
| 1 | 要点逐条匹配 | 标准要点 + 学生答案 | `step_scores[{point, score, comment}]` | 知识点覆盖度 |
| 2 | 整体逻辑评估 | 学生答案全文 | `logic_score` (0-5) + `logic_comment` | 推理连贯性与自洽性 |
| 3 | 综合反馈 | 步骤1+2 结果 | `total_score + feedback + weak_tags` | 综合质量与薄弱标记 |

### Русский текст

Основная логика оценивания системы реализована в методе `grade_answer()` модуля `cot_grader.py`. Данный метод использует технику промпт-инжиниринга Chain of Thought, принуждая LLM последовательно выполнять оценивание в три шага, каждый из которых выводит структурированные промежуточные результаты, обобщаемые в итоговую оценку. Такая пошаговая конструкция обеспечивает прозрачность, объяснимость и проверяемость процесса оценивания.

**Шаг 1: Последовательное сопоставление критериев (Point-wise Matching).** Система последовательно сопоставляет каждый критерий правильного ответа (`standard_answer_points`) с ответом студента. Для каждого критерия LLM определяет, охвачен ли он в ответе студента, степень охвата и выставляет оценку от 0 до 5 с комментарием. Результатом данного шага является массив `step_scores`, каждый элемент которого содержит `point_index` (индекс критерия), `student_content` (релевантная часть ответа студента), `score` (0-5) и `comment` (обоснование оценки). Максимальная общая сумма баллов вычисляется как количество критериев, умноженное на 5.

**Шаг 2: Общая оценка логики (Logic Assessment).** Помимо покрытия критериев, LLM выполняет независимую оценку общей логики ответа студента. Измеряемые аспекты включают: полноту цепочки рассуждений, непротиворечивость аргументации, наличие логических скачков или противоречий. Данный шаг выводит единый балл за логику (`logic_score`, 0-5) и соответствующий комментарий (`logic_comment`). Включение логического балла компенсирует невозможность модели «последовательного сопоставления» оценить качество аргументации в целом.

**Шаг 3: Комплексная обратная связь и маркировка слабых мест (Comprehensive Feedback).** Объединяя результаты первых двух шагов, LLM вычисляет итоговый балл (`total_score = sum(step_scores.score) + logic_score`) и генерирует трёхчастную обратную связь: краткая общая оценка, сократические наводящие вопросы и маркировка слабых мест (`weak_tags`). Теги записываются в таблицу `knowledge_tags` для последующего агрегированного анализа слабых мест.

В следующей таблице обобщены входные данные, выходные данные и измеряемые аспекты каждого шага цепочки рассуждений:

| Шаг | Название | Вход | Выход | Измеряемый аспект |
|-----|----------|------|-------|-------------------|
| 1 | Последовательное сопоставление | Критерии + ответ студента | `step_scores[{point, score, comment}]` | Полнота покрытия знаний |
| 2 | Общая оценка логики | Полный ответ студента | `logic_score` (0-5) + `logic_comment` | Связность и непротиворечивость |
| 3 | Комплексная обратная связь | Результаты шагов 1 и 2 | `total_score + feedback + weak_tags` | Общее качество + маркировка |

### 3.2.2 少样本提示 / Few-Shot Prompting

### 中文正文

为确保 LLM 输出格式的一致性和批改质量的稳定性，本系统在所有 LLM 调用中均采用少样本提示（Few-Shot Prompting）策略。每个 Prompt 模板中均嵌入一个完整的 JSON 输出格式示例，作为 LLM 输出格式的"视觉约束"。

在 `COT_GRADING_PROMPT` 中，示例输出包含一个虚构的批改案例——两个得分点、对应评分和评语、逻辑分、总分以及苏格拉底式反馈。LLM 在生成实际批改时会自动参考该示例的结构和格式。

少样本提示与 PydanticOutputParser 形成双重保障机制：前者通过示例引导 LLM 输出正确格式的 JSON，后者在代码层对 LLM 输出进行强制校验和类型转换。当 LLM 输出格式异常时，系统自动重试一次（重新请求并附加格式提醒），若仍失败则返回 None，由前端提示用户重试。

### Русский текст

Для обеспечения согласованности формата вывода LLM и стабильности качества оценивания система использует стратегию Few-Shot Prompting во всех вызовах LLM. В каждый шаблон промпта встраивается полный пример формата JSON-вывода, служащий «визуальным ограничением» для формата вывода LLM.

В `COT_GRADING_PROMPT` пример вывода содержит вымышленный случай оценивания с двумя критериями, соответствующими оценками и комментариями, баллом за логику, итоговым баллом и сократической обратной связью. При генерации реального оценивания LLM автоматически ориентируется на структуру и формат данного примера.

Few-Shot Prompting совместно с PydanticOutputParser образуют двойной механизм обеспечения: первый направляет LLM на генерацию JSON в правильном формате через пример, второй выполняет принудительную валидацию и преобразование типов на уровне кода. При аномальном формате вывода LLM система автоматически выполняет одну повторную попытку; в случае повторной неудачи возвращается None с предложением пользователю повторить попытку через интерфейс.

## 3.3 多角色权限管理 / Управление ролями и правами

本节阐述系统的用户权限体系：3.3.1 描述注册、登录、会话保持的完整认证流程及其安全设计；3.3.2 说明基于"数据层过滤 + 前端层隐藏"双重策略的角色隔离实现方式。

Данный раздел описывает систему прав доступа: в разделе 3.3.1 представлен полный процесс аутентификации (регистрация, вход, поддержание сессии) и его архитектура безопасности; в разделе 3.3.2 объясняется реализация изоляции ролей на основе двойной стратегии «фильтрация на уровне данных + сокрытие на уровне интерфейса».

### 3.3.1 认证流程 / Процесс аутентификации

### 中文正文

本系统的用户认证流程基于 `db.py` 提供的数据库操作和 Streamlit 的 `st.session_state` 会话状态管理实现。认证流程设计遵循"简洁但安全"原则——在满足毕设演示需求的前提下，实现基本的身份验证与角色区分。

**注册流程。** 用户在登录页面选择角色（教师/学生）后，切换至注册模式。表单收集用户名、密码和角色三项信息。前端进行空值检查和密码长度校验（不少于 3 个字符），通过后调用 `register_user()` 函数。该函数使用 `hashlib.sha256` 对密码进行哈希处理，将用户名、哈希值和角色存入 `users` 表。若用户名已存在，返回"用户名已存在"错误提示。密码哈希存储确保即使数据库泄露，原始密码也难以还原。

**登录流程。** 用户输入用户名和密码，系统调用 `login_user()` 函数，对输入的密码进行 SHA-256 哈希后与数据库比对。若匹配，返回用户信息字典并写入 `st.session_state`，包括 `user_id`、`username`、`role` 三个字段。若哈希不匹配或用户不存在，返回 None，前端显示"用户名或密码错误"提示。登录成功后，`st.session_state["role"]` 驱动侧边栏菜单的角色自适应切换，将用户导向对应的功能页面组。

**会话保持。** 一旦登录成功，用户信息在整个 Streamlit 应用会话期间持续可用。`st.session_state` 在 Streamlit 的脚本重执行之间保持状态，使得页面切换不会丢失用户登录信息。退出登录时，清除 `session_state` 中的用户相关字段并重定向至登录页。

### Русский текст

Процесс аутентификации пользователей в данной системе реализован на основе операций с базой данных из `db.py` и управления состоянием сессии Streamlit через `st.session_state`. Дизайн процесса следует принципу «простота при сохранении безопасности» — в рамках требований дипломного демо-прототипа обеспечивается базовая проверка личности и разделение ролей.

**Процесс регистрации.** Пользователь выбирает роль (преподаватель/студент) на странице входа и переключается в режим регистрации. Форма собирает три поля: имя пользователя, пароль и роль. Выполняется проверка на пустые значения и минимальную длину пароля (не менее 3 символов). После успешной проверки вызывается функция `register_user()`, которая хеширует пароль с помощью `hashlib.sha256` и сохраняет имя пользователя, хеш и роль в таблицу `users`. При повторном имени пользователя возвращается сообщение «имя пользователя уже существует». Хеширование пароля гарантирует, что даже при утечке базы данных исходные пароли восстановить практически невозможно.

**Процесс входа.** Пользователь вводит имя и пароль; система вызывает `login_user()`, хеширует введённый пароль SHA-256 и сравнивает с записью в базе данных. При совпадении возвращается словарь с данными пользователя, который записывается в `st.session_state` с тремя полями: `user_id`, `username`, `role`. При несовпадении хеша или отсутствии пользователя возвращается None, а интерфейс выводит сообщение «неверное имя пользователя или пароль». После успешного входа `st.session_state["role"]` управляет адаптивным переключением меню боковой панели, направляя пользователя к соответствующей группе функциональных страниц.

**Сохранение сессии.** После входа информация о пользователе остаётся доступной на протяжении всей сессии приложения Streamlit. `st.session_state` сохраняет состояние между перезапусками скрипта Streamlit, поэтому переключение страниц не приводит к потере данных о пользователе. При выходе выполняется очистка полей пользователя из `st.session_state` и перенаправление на страницу входа.

## 3.4 实时聊天系统 / Система чата в реальном времени

本节介绍师生即时通讯模块的实现细节：3.4.1 解析基于短轮询与 fragment 局部刷新的准实时架构，涵盖消息存储、发送流程和 UI 渲染；3.4.2 分类说明系统支持的图片、视频、文档、压缩包四种文件类型的传输与预览方式。

Данный раздел описывает детали реализации модуля мгновенной коммуникации: в разделе 3.4.1 анализируется квази-реальная архитектура на основе короткого опроса и частичного обновления fragment, охватывая хранение сообщений, процесс отправки и рендеринг UI; в разделе 3.4.2 классифицируются способы передачи и предпросмотра четырёх типов файлов — изображений, видео, документов и архивов.

### 3.4.1 技术实现 / Техническая реализация

### 中文正文

师生聊天系统是本系统中最具交互性的功能模块。在技术选型上，由于 Streamlit 是基于"脚本重执行"模型的同步框架，不支持 WebSocket 等真正的全双工通信协议，本系统采用"短轮询（Short Polling）+ 局部刷新"的方案实现准实时消息传递。

**消息存储模型。** 所有聊天消息存储在 SQLite 的 `messages` 表中。每条消息包含 `sender_id`（发送者）、`receiver_id`（接收者）、`content`（文本内容）、`file_path/file_name/file_type`（附件信息）和 `created_at`（时间戳）。消息的收发双方由 `sender_id` 和 `receiver_id` 的组合唯一确定。查询时通过 `get_messages(user_id_1, user_id_2)` 获取两个用户之间的全部消息历史记录。

**准实时轮询机制。** 聊天界面中的消息展示区域通过 Streamlit 的 `@st.fragment(run_every=2)` 装饰器实现每 2 秒自动局部刷新。该机制仅在聊天页面处于活跃焦点时才执行，不会触发整页重渲染，因此对其他页面的性能没有影响。每次刷新时，前端请求最新的消息列表并重新渲染聊天气泡。

**消息发送流程。** 用户通过底部的输入框输入文本，或通过文件上传器选择附件，点击发送按钮后触发 `on_send_message()` 回调函数。该回调调用 `send_message()` 写入数据库。文件上传使用 `st.file_uploader` 组件，文件保存至 `uploads/chat/` 目录，文件名格式为"时间戳_原始文件名"，避免命名冲突。

**UI 渲染。** 聊天界面通过 HTML + CSS 实现微信风格的消息气泡。通过 `st.markdown(unsafe_allow_html=True)` 注入自定义样式，发送者消息靠右（蓝色气泡），接收者消息靠左（灰色气泡），消息上方显示发送者名字和时间戳。图片、视频和文件附件以不同的 HTML 标签渲染。

### Русский текст

Система чата между преподавателем и студентом является наиболее интерактивным функциональным модулем. С точки зрения выбора технологии, поскольку Streamlit — это синхронный фреймворк, работающий по модели «перезапуска скрипта» и не поддерживающий истинные двунаправленные протоколы связи, такие как WebSocket, данная система использует подход «короткого опроса (Short Polling) + частичного обновления» для обеспечения квази-реальной передачи сообщений.

**Модель хранения сообщений.** Все сообщения чата хранятся в таблице `messages` базы данных SQLite. Каждое сообщение содержит `sender_id` (отправитель), `receiver_id` (получатель), `content` (текст), `file_path/file_name/file_type` (вложения) и `created_at` (временная метка). Пары отправитель-получатель однозначно определяются комбинацией `sender_id` и `receiver_id`. История сообщений между двумя пользователями запрашивается через `get_messages(user_id_1, user_id_2)`.

**Механизм квази-реального опроса.** Область отображения сообщений в интерфейсе чата использует декоратор Streamlit `@st.fragment(run_every=2)` для автоматического частичного обновления каждые 2 секунды. Данный механизм срабатывает только когда страница чата находится в активном фокусе, не вызывая полного перерендеринга страницы и не влияя на производительность других страниц. При каждом обновлении интерфейс запрашивает актуальный список сообщений и перерендеривает пузырьки чата.

**Процесс отправки сообщений.** Пользователь вводит текст в поле ввода или выбирает вложение через загрузчик файлов, нажимает кнопку отправки, что запускает callback-функцию `on_send_message()`. Callback вызывает `send_message()` для записи в базу данных. Файлы загружаются через компонент `st.file_uploader` и сохраняются в каталог `uploads/chat/` с именем в формате «временная_метка_оригинальное_имя_файла» для предотвращения конфликтов имён.

**Рендеринг UI.** Интерфейс чата реализует пузырьки сообщений в стиле WeChat с помощью HTML + CSS. Через `st.markdown(unsafe_allow_html=True)` внедряются пользовательские стили: сообщения отправителя выравниваются вправо (синий пузырёк), сообщения получателя — влево (серый пузырёк), над сообщением отображаются имя отправителя и временная метка. Изображения, видео и файлы рендерятся различными HTML-тегами.

### 3.4.2 支持的文件类型 / Поддерживаемые типы файлов

### 中文正文

聊天系统支持多种文件类型的传输和在线预览，覆盖教学场景中的常见文件格式。系统根据文件扩展名自动识别类型并采用不同的渲染方式。

**图片格式（png, jpg, jpeg, gif, webp）。** 图片文件在聊天气泡下方以内联方式直接预览，使用 `<img>` 标签渲染，固定最大宽度为 300px 以保持界面整洁。用户点击图片可查看原图。

**视频格式（mp4, mov, avi）。** 视频文件使用 Streamlit 原生 `st.video` 组件渲染，支持播放控制（播放/暂停、进度条、音量调节），无需离开聊天页面即可观看。

**文档格式（pdf, docx, txt, pptx）。** 文档文件以链接形式展示，显示文件名和文件类型图标。用户点击链接后在浏览器新标签页中打开或下载。系统不对文档内容进行在线预览，以避免解析复杂性和安全性问题。

**压缩包格式（zip, rar）。** 与文档格式相同，以文件名链接方式展示，支持下载。

下表总结了各类文件的处理方式：

| 文件类型 / Тип | 扩展名 / Расширения | 渲染方式 / Способ рендеринга | 预览支持 / Предпросмотр |
|--------------|-------------------|---------------------------|---------------------|
| 图片 | png, jpg, gif, webp | `<img>` 内联显示 (max-width: 300px) | 直接预览 |
| 视频 | mp4, mov, avi | `st.video` 播放器 | 在线播放 |
| 文档 | pdf, docx, txt, pptx | 文件链接（`<a>` 标签） | 下载/新标签页打开 |
| 压缩包 | zip, rar | 文件链接（`<a>` 标签） | 下载 |

### Русский текст

Система чата поддерживает передачу и онлайн-просмотр файлов различных типов, охватывая распространённые форматы, используемые в образовательных сценариях. Система автоматически определяет тип файла по его расширению и применяет соответствующий способ рендеринга.

**Изображения (png, jpg, jpeg, gif, webp).** Изображения отображаются непосредственно под пузырьком сообщения с помощью тега `<img>` с фиксированной максимальной шириной 300px для сохранения чистоты интерфейса. Пользователь может нажать на изображение для просмотра в полном размере.

**Видео (mp4, mov, avi).** Видеофайлы рендерятся с помощью нативного компонента Streamlit `st.video` с поддержкой управления воспроизведением (воспроизведение/пауза, полоса прогресса, регулировка громкости) без необходимости покидать страницу чата.

**Документы (pdf, docx, txt, pptx).** Документы отображаются в виде ссылок с именем файла и иконкой типа файла. Пользователь может открыть документ в новой вкладке браузера или скачать его по ссылке. Система не выполняет онлайн-предпросмотр содержимого документов во избежание сложностей синтаксического анализа и проблем безопасности.

**Архивы (zip, rar).** Аналогично документам, отображаются в виде ссылок с именем файла с возможностью скачивания.

В следующей таблице обобщены способы обработки каждого типа файлов:

| Тип файла | Расширения | Способ рендеринга | Предпросмотр |
|-----------|-----------|-------------------|-------------|
| Изображение | png, jpg, gif, webp | `<img>` встроенный (max-width: 300px) | Прямой просмотр |
| Видео | mp4, mov, avi | Проигрыватель `st.video` | Онлайн-воспроизведение |
| Документ | pdf, docx, txt, pptx | Ссылка на файл (`<a>`) | Скачивание/новая вкладка |

## 4.1 页面导航架构 / Архитектура навигации по страницам

本节从交互入口和导航结构两个维度分析系统的页面架构：4.1.1 展示登录/注册页面的角色选择与表单设计；4.1.2 展示侧边栏导航的角色自适应菜单与页面切换机制。

### 4.1.1 登录/注册页 / Страница входа/регистрации

### 中文正文

登录/注册页是系统的唯一入口页面，承担用户身份验证和角色分流的核心功能。页面布局分为三个区域：

**顶部：角色选择区。** 用户通过水平排列的单选按钮（`st.radio`）在"教师"和"学生"两个角色之间选择。角色选择决定了后续注册字段中 `role` 参数的取值，也影响登录成功后进入的功能页面组。该选择在登录/注册模式下均可用。

**中部：登录/注册表单区。** 页面通过 `st.toggle` 或按钮切换登录与注册两种操作模式。在登录模式下，表单包含用户名和密码两个输入框；在注册模式下，表单额外显示角色标签（由顶部角色选择决定）和密码确认框。表单提交前进行空值检查和密码长度（≥3 个字符）校验。注册成功后，数据库写入用户信息，同时页面提示"注册成功，请切换至登录"。登录成功后，用户信息写入 `st.session_state`，页面自动跳转至对应的功能页。

**底部：状态提示区。** 显示操作结果反馈——错误信息（如"用户名已存在"、"用户名或密码错误"）以 `st.error` 红色警示框展示，成功信息以 `st.success` 绿色提示框展示。所有提示在用户下一次交互后自动消除。

### Русский текст

Страница входа/регистрации является единственной входной точкой системы, выполняющей ключевые функции аутентификации пользователей и распределения по ролям. Макет страницы разделён на три области:

**Верхняя часть: выбор роли.** Пользователь выбирает между ролями «преподаватель» и «студент» с помощью горизонтальных переключателей (`st.radio`). Выбор роли определяет значение параметра `role` в форме регистрации, а также набор функциональных страниц, доступных после входа. Данный выбор доступен как в режиме входа, так и в режиме регистрации.

**Средняя часть: форма входа/регистрации.** Страница использует `st.toggle` или кнопки для переключения между режимами входа и регистрации. В режиме входа форма содержит поля для имени пользователя и пароля; в режиме регистрации дополнительно отображается метка роли и поле подтверждения пароля. **Нижняя часть: область статуса.** Отображает результаты операций: сообщения об ошибках (например, «имя пользователя уже существует», «неверное имя пользователя или пароль») выводятся в красном предупреждающем блоке `st.error`, а сообщения об успехе — в зелёном блоке `st.success`. Все подсказки автоматически исчезают после следующего взаимодействия пользователя.

### 4.1.2 侧边栏导航 / Боковая панель навигации

### 中文正文

## 4.3 学生端交互流程 / Пользовательский поток студента

本节展示学生端的交互路径——从查看习题、提交作答、获取批改反馈，到查看成绩趋势和错题回顾。

Данный раздел демонстрирует путь взаимодействия студента — от просмотра заданий, отправки ответов и получения обратной связи до просмотра динамики оценок и повторения ошибок.

## 第五章：后续改进方案 / Глава 5: План дальнейших улучшений

### 中文正文

**代码层改进。** 当前系统的密码存储采用 SHA-256 哈希，后续应迁移至 bcrypt + salt，以抵御彩虹表攻击。前端 Streamlit 框架虽利于快速原型开发，但在精细 UI 控制和实时通信方面存在局限，可向 FastAPI + Vue/React 前后端分离架构演进。同时，应将 main.py（约 726 行）按职责拆分为独立页面模块，引入依赖注入模式管理 RAGEngine 和 CoTGrader 的单例生命周期，并补充单元测试覆盖。

**实现层改进。** 聊天模块当前采用 2 秒轮询（`st.fragment run_every=2`）模拟实时通信，后续可替换为 FastAPI WebSocket 推送，实现零延迟传输并减少不必要的数据库查询。在模型层面，可接入 DeepSeek-V3、GLM-4、GPT-4o、Claude 等多模型进行横向对比，覆盖出题质量、批改准确率、格式遵循率、响应速度等维度，并使用 RAGAS 框架（Context Relevance + Faithfulness）进行自动化评测。检索策略可从当前的纯 Dense Retrieval 升级为 Dense + Sparse 混合检索（BM25 + Embedding），提升关键词匹配的召回率。知识点追踪方面，应从基于标签的简单计数升级为 BKT（Bayesian Knowledge Tracing）或 DKT（Deep Knowledge Tracing），动态评估学生知识掌握概率。

**交互层与系统评估改进。** 当前界面仅适配桌面端，后续需增加响应式布局以支持手机和平板。应补充通知系统——新习题发布、批改完成、新消息到来时均推送实时提醒。界面语言方面可扩展中/俄/英三语切换。系统评估方面，计划从自动化评测和人工评测两个维度展开：自动化层面使用 RAGAS 框架量化评估 RAG 管道的质量；人工层面收集 30-50 份批改样本，采用 Likert 量表（1-5 分）评估连贯性与准确性，通过 Pearson 相关系数和 MSE 进行对比分析，并对典型案例进行深度拆解，验证系统的实际教学效果。

### Русский текст

**Улучшения уровня кода.** В текущей системе для хранения паролей используется SHA-256, который следует заменить на bcrypt + salt для защиты от атак по радужным таблицам. Фреймворк Streamlit удобен для быстрого прототипирования, но имеет ограничения в тонком контроле UI и работе с real-time коммуникацией — целесообразна миграция на архитектуру FastAPI + Vue/React с разделением фронтенда и бэкенда. Кроме того, main.py (~726 строк) следует разделить на независимые модули страниц по принципу единственной ответственности, внедрить паттерн Dependency Injection для управления синглтонами RAGEngine и CoTGrader, а также добавить модульные тесты.

**Улучшения уровня реализации.** Модуль чата в настоящее время использует опрос каждые 2 секунды (`st.fragment run_every=2`) для имитации real-time связи. В перспективе он должен быть заменён на WebSocket-пушинг через FastAPI, что обеспечит нулевую задержку и сократит избыточные запросы к базе данных. На уровне моделей планируется подключить DeepSeek-V3, GLM-4, GPT-4o, Claude для сравнительного анализа по таким метрикам, как качество генерации заданий, точность оценивания, соблюдение формата и скорость ответа, а также использовать фреймворк RAGAS (Context Relevance + Faithfulness) для автоматизированной оценки. **Улучшения уровня взаимодействия и оценка системы.** Текущий интерфейс ориентирован только на десктопы — требуется добавить адаптивную вёрстку для планшетов и мобильных устройств. Необходимо внедрить систему уведомлений: оповещения о новых заданиях, завершении проверки и новых сообщениях. Интерфейс также может быть расширен переключением между китайским, русским и английским языками. Оценка системы планируется по двум направлениям: автоматизированная (RAGAS для количественной оценки пайплайна RAG) и экспертная (30-50 образцов проверок с оценкой по шкале Лайкерта 1-5, расчёт коэффициента корреляции Пирсона и MSE). Глубокий анализ типовых кейсов позволит верифицировать практическую эффективность системы в учебном процессе.

## 结论 / Заключение

### 中文正文

本文设计并实现了一套基于 RAG 与 CoT 的智能教学评估系统，围绕"知识锚定"与"可解释评估"两个核心问题，完成了从底层模型调用到上层用户交互的全链路构建。在代码层面，系统以 Python + Streamlit 实现快速原型交付，通过 SQLite 与 ChromaDB 构建轻量级数据层，结合 DeepSeek API 实现语言智能服务。在实现层面，RAG 引擎将 LLM 输出锚定于教材内容，有效抑制幻觉；CoT 批改引擎通过三步推理链输出结构化评分，提供透明的可解释反馈；师生聊天模块实现了课后的即时沟通。在交互层面，系统支持教师-学生双角色隔离的多页面架构，教师可完成教材管理、习题生成与发布、成绩统计，学生可答题、查看批改反馈、追踪薄弱知识点。整套系统代码量约 1400 行，从零开发至原型交付历时 3 天，验证了轻量级技术路线在教育 AI 原型研究中的可行性。后续工作将围绕 WebSocket 实时通信、多模型对比评测、RAGAS 自动化评估、混合检索策略和知识追踪模型等方向展开，进一步提升系统的实用性和评估深度。

### Русский текст

В данной работе спроектирована и реализована интеллектуальная система оценки обучения на основе RAG и CoT, ориентированная на два ключевых аспекта: «привязка знаний» и «объяснимое оценивание». Система охватывает полный конвейер от вызова базовых моделей до пользовательского взаимодействия верхнего уровня. На уровне кода система использует Python + Streamlit для быстрой поставки прототипа, SQLite и ChromaDB для лёгкого слоя данных, а также DeepSeek API для языковых интеллектуальных сервисов. На уровне реализации движок RAG привязывает вывод LLM к содержанию учебных материалов, эффективно подавляя галлюцинации; движок оценивания CoT выводит структурированные оценки через трёхшаговую цепочку рассуждений, обеспечивая прозрачную и объяснимую обратную связь; модуль чата преподавателя и студента реализует оперативную коммуникацию после занятий. На уровне взаимодействия система поддерживает многопользовательскую архитектуру с разделением ролей преподавателя и студента: преподаватель может управлять материалами, генерировать и публиковать задания, просматривать статистику; студент — отвечать на задания, просматривать результаты проверки и отслеживать слабые темы. Общий объём кода системы составляет около 1400 строк, разработка от нуля до готового прототипа заняла 3 дня, что подтверждает жизнеспособность лёгкой технологической траектории в исследованиях образовательных AI-прототипов. Дальнейшая работа будет направлена на WebSocket-коммуникацию в реальном времени, сравнительное тестирование нескольких моделей, автоматизированную оценку RAGAS, гибридные стратегии поиска и модели отслеживания знаний для повышения практической применимости и глубины оценки системы.
